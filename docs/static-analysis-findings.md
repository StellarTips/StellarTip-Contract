## Medium Severity

### M-1: Unchecked `unwrap()` on storage read

**File:** `src/lib.rs:452`

```rust
let fee_recipient: Address =
    env.storage().instance().get(&DataKey::FeeRecipient).unwrap();
```

**Description:** The `FeeRecipient` storage key is read with a bare `.unwrap()` instead of `unwrap_or_else(|| panic_with_error!(...))`. If the contract is somehow initialized without a fee recipient (or storage is corrupted), this will panic with a generic message rather than a typed error.

**Recommendation:** Replace with `unwrap_or_else` and a proper `TipError` variant:

```rust
let fee_recipient: Address = env
    .storage()
    .instance()
    .get(&DataKey::FeeRecipient)
    .unwrap_or_else(|| panic_with_error!(env, TipError::InvalidInput));
```

### M-2: Potential unbounded iteration in `get_tips`

**File:** `src/lib.rs:590-603`

```rust
pub fn get_tips(env: Env, creator: Address, start: u64, limit: u64) -> Vec<Tip> {
    let mut results = Vec::new(&env);
    let count: u64 =
        env.storage().persistent().get(&DataKey::TipCount(creator.clone())).unwrap_or(0);
    let end = (start + limit).min(count);
    for i in start..end {
        // ...
    }
    results
}
```

**Description:** If a caller requests a large `limit`, this function iterates over many storage entries in a single call. While capped by `TipCount`, this could lead to resource exhaustion (instruction limit) if the creator has a large tip history.

**Recommendation:** Add an explicit maximum limit (e.g., 100):

```rust
const MAX_TIP_LIMIT: u64 = 100;
// ...
let limit = limit.min(MAX_TIP_LIMIT);
```

## Low Severity

### L-1: Generic panic in `unwrap_or_else` closures

**File:** `src/lib.rs` (multiple locations: 226, 241, 258, 272, 275, 292, 295, 360, 383)

**Description:** Several `unwrap_or_else` calls use `panic_with_error!` with typed errors, which is correct. However, a few storage reads (e.g., `get(&DataKey::Admin)`) could panic with a less descriptive error if the admin key is unexpectedly missing. The existing error handling is adequate for normal operation but could be more explicit.

**Recommendation:** No action required — these are guarded by initialization checks. Monitor if any false positives appear in CI.

### L-2: Integer casting in fee calculation

**File:** `src/lib.rs:442`

```rust
let fee = (amount * (fee_bps as i128)) / (MAX_FEE_BPS as i128);
```

**Description:** The fee calculation casts `fee_bps` (u32) and `MAX_FEE_BPS` (u32) to `i128`. In theory, the multiplication `amount * fee_bps` could overflow if `amount` were extremely large, though i128 provides 128 bits of range, making this impractical. Scout's "divide-before-multiply" detector may flag the multiplication order.

**Recommendation:** No action required. The multiplication-before-division order is correct for precision. The i128 type provides sufficient range.
